## 违背程度排序（由高到低）

### 🔴 严重违背（违反核心原则）

#### 1. goroutine 泄漏问题
- **类别**: Quality 类别
- **问题**: dispatcher.go 超时后 goroutine 未正确退出，导致资源泄漏
- **违背原则**: ❌ **More Action** - 资源泄漏限制了 Agent 的持续行动能力
- **违背程度**: 🔴 严重 - 长期运行可能耗尽资源，完全阻止 Agent 行动
- **已修复**: ✅ 添加 timeoutCtx.Done() 检查

#### 2. 返回零值而非错误
- **类别**: Quality 类别
- **问题**: state_store.go Load 方法返回零值而非错误，语义不清
- **违背原则**: ❌ **More Context** - 减少了上下文信息，调用者无法区分"不存在"和"存在但为空"
- **违背程度**: 🔴 严重 - 直接违反 More Context 核心原则
- **已修复**: ✅ 改为返回明确错误

#### 3. 上下文信息不一致
- **类别**: Documentation 类别
- **问题**: 文档更新滞后于代码变更，导致文档与代码不一致
- **违背原则**: ❌ **More Context** - 上下文信息不一致，误导 Agent 决策
- **违背程度**: 🔴 严重 - 不一致的上下文可能导致错误决策
- **已修复**: ✅ 修复后立即更新 HANDOVER.md

### 🟠 中度违背（限制行动能力）

#### 4. channel 重复关闭
- **类别**: Quality 类别
- **问题**: lifecycle.go done channel 缺少单次执行保护，重复关闭导致 panic
- **违背原则**: ❌ **More Action** - 缺乏保护限制了 Agent 的行动能力
- **违背程度**: 🟠 中度 - panic 导致系统崩溃，阻止所有行动
- **已修复**: ✅ 使用 sync.Once 保护

#### 5. 循环依赖检测算法错误
- **类别**: Quality 类别
- **问题**: scheduler.go visited map 状态管理不当，循环依赖检测失效
- **违背原则**: ⚠️ **More Action** - 算法错误限制了 Agent 的正确行动
- **违背程度**: 🟠 中度 - 可能导致死锁，间接限制行动
- **已修复**: ✅ 修正 visited map 管理

#### 6. 接口变更管理
- **类别**: Quality 类别
- **问题**: GetStatus 签名变更时需要强制更新所有调用点
- **违背原则**: ⚠️ **Zero Control** - 可能限制了 Agent 的灵活性
- **违背程度**: 🟠 中度 - 强制变更可能限制 Agent 的自主性
- **已修正**: ✅ 改为建议性提醒，而非强制检查
