# 未使用内容修复风险评估

**评估日期**: 2026-05-08
**评估目的**: 分析修复未使用内容的潜在风险

---

## 1. cd-workflow.yml: sign-artifacts job

### 当前状态
- 位置: cd-workflow.yml lines 114-141
- 功能: 使用 GPG 签名二进制文件
- 问题: 生成签名但未在 create-release job 中使用

### 修复选项

#### 选项 A: 在 create-release 中使用签名
**操作**: 在 create-release job 中下载并上传签名文件

**潜在风险**:
- ⚠️ **中等风险**: 如果 GPG 密钥配置不正确，签名可能失败
- ⚠️ **中等风险**: 签名验证可能失败（用户端验证）
- ⚠️ **低风险**: 增加发布流程复杂度
- ⚠️ **低风险**: 增加 CI/CD 时间

**缓解措施**:
- 在测试环境中验证签名流程
- 提供签名验证指南
- 监控签名失败率

**建议**: 如果项目需要代码签名，实施此选项

#### 选项 B: 删除 sign-artifacts job
**操作**: 完全删除 sign-artifacts job

**潜在风险**:
- ❌ **高风险**: 如果用户依赖签名验证，删除后无法验证
- ❌ **高风险**: 安全性降低（无法验证二进制完整性）
- ⚠️ **中等风险**: 已发布的版本可能有签名，新版本没有，不一致
- ⚠️ **低风险**: 无法回滚到有签名的版本

**缓解措施**:
- 确认用户不依赖签名验证
- 在发布说明中说明不再提供签名
- 保留签名密钥以便将来恢复

**建议**: 如果项目不需要代码签名，实施此选项

### 风险总结
- **修复风险**: 中等到高
- **建议**: 先确认项目是否需要代码签名
- **安全考虑**: 删除签名会降低安全性

## 2. ci.yml: docs job

### 当前状态
- 位置: ci.yml lines 143-167
- 功能: 生成 API 文档（go doc -all）
- 问题: 生成文档但未在其他地方使用

### 修复选项

#### 选项 A: 上传到 GitHub Pages
**操作**: 使用 peaceiris/actions-gh-pages@v3 上传文档

**潜在风险**:
- ⚠️ **低风险**: GitHub Pages 配置可能失败
- ⚠️ **低风险**: 文档可能与代码不同步
- ⚠️ **低风险**: 增加发布时间
- ⚠️ **低风险**: 需要配置 GitHub Pages

**缓解措施**:
- 在单独的分支管理 GitHub Pages
- 设置自动化部署
- 监控文档生成失败率

**建议**: 如果项目需要公开文档，实施此选项

#### 选项 B: 删除 docs job
**操作**: 完全删除 docs job

**潜在风险**:
- ⚠️ **中等风险**: 失去自动化文档生成
- ⚠️ **低风险**: 文档可能过时
- ⚠️ **低风险**: 开发者需要手动生成文档
- ✅ **无风险**: 不影响代码质量

**缓解措施**:
- 在本地使用 go doc 生成文档
- 定期手动更新文档
- 在 README 中提供文档生成命令

**建议**: 如果不需要自动化文档，实施此选项

### 风险总结
- **修复风险**: 低
- **建议**: 删除 docs job（当前里程碑不需要自动化文档）
- **影响**: 轻微（可以手动生成文档）

## 3. release.yml 与 cd-workflow.yml 重复

### 当前状态
- release.yml: 简单的发布流程（仅构建二进制）
- cd-workflow.yml: 完整的发布流程（多平台、Docker、签名）
- 问题: 功能重复，维护两套流程

### 修复选项

#### 选项 A: 删除 release.yml
**操作**: 完全删除 release.yml，统一使用 cd-workflow.yml

**潜在风险**:
- ⚠️ **中等风险**: 如果有其他系统依赖 release.yml，会失败
- ⚠️ **中等风险**: cd-workflow.yml 可能更复杂，故障点更多
- ⚠️ **低风险**: 需要更新文档和脚本引用
- ⚠️ **低风险**: 团队需要适应新的发布流程

**缓解措施**:
- 搜索所有引用 release.yml 的地方
- 提供迁移指南
- 在废弃期间保留 release.yml 一段时间
- 设置监控，确保 cd-workflow.yml 正常工作

**建议**: 实施此选项，统一发布流程

#### 选项 B: 保留 release.yml 作为快速发布
**操作**: 保留 release.yml 用于简单发布，cd-workflow.yml 用于完整发布

**潜在风险**:
- ⚠️ **低风险**: 维护两套流程增加复杂度
- ⚠️ **低风险**: 可能混淆使用场景
- ⚠️ **低风险**: 两套流程可能不一致

**缓解措施**:
- 明确文档说明两个流程的使用场景
- 定期同步两个流程的配置
- 考虑废弃 release.yml

**建议**: 不推荐，增加维护成本

#### 选项 C: 重命名 release.yml 为 legacy-release.yml
**操作**: 重命名并标记为废弃

**潜在风险**:
- ⚠️ **低风险**: 可能仍有系统引用
- ⚠️ **低风险**: 废弃版本可能被误用

**缓解措施**:
- 添加明确的废弃警告
- 在文档中说明废弃原因
- 设置一段时间后删除

**建议**: 过渡方案，最终删除

### 风险总结
- **修复风险**: 低到中等
- **建议**: 删除 release.yml，统一使用 cd-workflow.yml
- **影响**: 中等（需要更新引用）
