## 一、第一性原理：Axis 原生调度的元定义

传统调度系统的元假设是**"任务是被调度的客体"**。Axis 的原生元假设是：

> **Agent 是自主的意图体，调度是意图的协调协议，不是工作的分配机制。**

由此导出五个不可约简的公理：

1. **自主公理**：Agent 拥有本地决策权，调度器只提供认知环境，不 micromanage。
2. **动态公理**：执行路径在运行时涌现，非预先编排。
3. **不确定公理**：任务完成度是概率分布，不是布尔状态。
4. **协同公理**：多 Agent 协作是能力互补的动态组队，不是流程节点的串联。
5. **生命周期公理**：Agent 存在出生、成长、分化、休眠、传承的有机过程，不是 CRUD。

**所有场景均从这五条公理推演，拒绝从 BPMN、Cron、RPA、DAG 等传统范式反向映射。**

## 三、通用基础层：意图的直接兑现

### 3.1 个人开发者

- **适用角色**：独立黑客、全栈开发者、AI 应用原型师、技术写作者
- **本质痛点**：个人认知带宽有限，上下文在 IDE/浏览器/终端/文档间碎片化，单线程执行无法匹配并行思考
- **核心价值**：将自然语言意图直接编译为自主执行体，**一人即拥有可并行的虚拟协作网络**。个人开发者不再是被工具链服务，而是拥有一个可扩展的"外接认知皮层"。

*Axis 原生映射*：`axis ask` → 意图解析 → AgentTask → 本地运行时执行 → 上下文预检。开发者无需理解 orchestrator、provider、scheduler 的分层，只需表达意图。

### 3.2 AI 原生创业团队

- **适用角色**：2–10 人 AI 创业公司，零传统技术债，产品即 Agent 系统
- **本质痛点**：需要快速验证多 Agent 产品形态，但业界缺乏"Agent 原生"的底层基建，被迫用传统后端+LLM 胶水拼凑
- **核心价值**：Axis 提供从零生长的原生骨架——**不是"在现有架构上加 AI"，而是"以 Agent 为原子的架构"**。创业团队的每一次产品迭代，都在强化 Agent 协作协议本身。

*Axis 原生映射*：本地控制平面 P0 已支持跨进程任务提交与状态查询，为创业团队提供"先单机验证，后集群扩展"的连续体。

## 四、高阶复杂层：集群的语义治理

### 4.1 企业中后台

- **适用角色**：企业平台架构组、AI 中台团队、数字化转型办公室
- **本质痛点**：传统 RPA 僵硬（任何变体即崩溃），现有 LLM 应用是提示词+胶水代码的堆砌，缺乏 Agent 级抽象；审批流、合规检查、数据搬运以"流程"而非"意图"建模
- **核心价值**：用 **Agent 合约** 替换流程引擎，实现"意图驱动"而非"流程驱动"。Agent 携带自身能力描述和约束边界进入系统，调度器基于语义匹配而非硬编码路由。

*Axis 原生映射*：M2 合约准入（contract admission）将企业合规规则编码为准入合约，Agent 提交任务前自动验证合约兼容性，失败即反馈而非运行时崩溃。

### 4.2 多 Agent 集群运维

- **适用角色**：SRE、AI Infra 团队、Agent 平台运维工程师
- **本质痛点**：当 Agent 数量从 1 增长到 100，手动管理生命周期不可持续；Agent 异构（有的重推理、有的重工具调用、有的长时休眠），传统容器编排无法感知语义能力
- **核心价值**：**语义化集群治理**——Agent 自主注册能力画像，调度器基于实时负载、上下文局部性、能力互补度进行动态分组；故障 Agent 自动隔离，能力缺失自动触发"招募"（唤醒备用 Agent）。

*Axis 原生映射*：M2 DAG 就绪集调度 + 优先级/容错机制，已从"任务队列"进化为"Agent 能力市场"。

### 4.3 行业垂直业务

- **适用角色**：金融、医疗、法律、制造的 AI 解决方案团队
- **本质痛点**：行业知识重、合规约束强、决策路径存在大量语义模糊性，传统自动化必须将模糊性硬编码为分支判断，维护成本极高
- **核心价值**：将行业规范编码为 **Agent 约束空间**（非规则引擎，而是可执行边界），Agent 在约束内拥有自主决策权。例如：医疗 Agent 知道"诊断建议必须经双人复核"，但具体如何获取复核是自主协调的。

*Axis 原生映射*：Adaptive Context Assembly 的 rule-based assembler 已为行业规则预留扩展点；Execution Context Summary 让行业审计可追溯到 Agent 决策时的上下文状态。

## 五、未来前瞻层：组织的智能重构

### 5.1 自动化办公（Agent-Native Office）

- **适用角色**：未来企业组织、DAO、自管理型团队、混合人机项目组
- **本质痛点**：组织知识沉淀在静态文档（PDF/Confluence/邮件）中，而非可执行体；人机协作边界由"谁使用工具"定义，而非"谁拥有意图"
- **核心价值**：办公场景中的 Agent **不是工具，而是具有岗位认知的数字同事**。它们拥有岗位描述（JD）、绩效上下文、交接班机制。人类从"操作工具"升级为"管理数字同事"，组织从"人员编制"进化为"能力编制"。

*Axis 原生映射*：沙盒化演进协议（Sandboxed Evolution Protocol）为"数字同事的能力成长"提供隔离实验-验证-晋升-废弃的完整生命周期管理。

### 5.2 数字员工集群（Digital Workforce Cluster）

- **适用角色**：未来人力资源平台、AI 劳动力市场、企业 CIO/CHRO
- **本质痛点**："AI 替代人"的叙事导致对抗性焦虑，根源在于缺乏"AI 作为劳动力"的管理范式；现有管理框架全是围绕人类员工设计的
- **核心价值**：Agent 拥有**简历（能力画像）、绩效记录（历史执行轨迹）、职业路径（能力分化树）、传承机制（知识蒸馏）**。Axis 提供的不只是调度，而是**数字劳动力的原生操作系统**——从招募（实例化）、培训（沙盒演进）、上岗（合约准入）、协作（上下文共享）到退休（归档传承）。

*Axis 原生映射*：这是 Axis 远期终极形态。当前 P0 的 ReadinessRegistry（就绪记录）是数字员工"在岗状态"的雏形；Preflight（预检）是"出勤检查"；Execution Context Summary 是"工作交接单"。

## 八、结语：Axis 不是调度系统，是 Agent 社会的基建协议

传统调度系统回答的问题是：**"如何把任务分给执行者？"**

Axis 回答的问题是：**"如何让自主的意图体在共享环境中有效协作？"**

这不是修辞差异，是元假设的彻底翻转。Axis 的终极产品形态不是更高效的"工作流引擎"，而是**Agent 社会的原生操作系统**——一个让智能体能够出生、协作、进化、传承的语义化环境。

M1–M6 完成了“一个人如何拥有自己的 Agent”和“Agent 如何自我评判与可控演化”。  
下一阶段将回答“一群 Agent 如何自治协作”和“Agent 如何作为数字生命体持续演化”。

这就是从第一性原理推演出的原生场景全景。
