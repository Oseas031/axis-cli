# Decision Record: Language Choice & Structural Tensions

**Date**: 2026-05-13
**Status**: Sublated (结论已沉淀)

## Phase I: Objectification（对象化）

### A0 Posture — 方向

两个问题的外化：
1. Go 是否是 Axis 的最优语言选择？
2. Axis 是否存在底层结构性缺陷？

### A1 Externalize — 外化

**语言选择的候选对象：**

| 语言 | 核心优势 | 核心劣势（对 Axis） |
|------|----------|-------------------|
| Go | 单二进制、goroutine 并发、快速编译、认知简洁 | 类型系统表达力有限、错误处理冗余 |
| TypeScript | 原型速度快、LLM SDK 生态、类型推断 | 需要运行时、启动慢、并发模型弱、无单二进制 |
| Rust | 零成本抽象、编译期安全、无 GC | 迭代慢 2-3x、async 心智负担、Agent 难以自修改 |
| Python | 生态最丰富、AI/ML 原生 | 性能差、无单二进制、类型松散、并发弱 |

**识别出的结构性张力：**

1. 自演化的验证悖论（谁验证验证者）
2. 单机假设的天花板
3. Context Assembly 的可扩展性
4. Competence 度量的 Goodhart's Law 风险
5. 缺乏内核级隔离层

## Phase II: Determinateness（规定性）

### A2 Inventory — 盘点已有规定

Axis 的核心约束条件（从设计原则推导）：

- **CLI First** → 必须单二进制、低启动延迟、无运行时依赖
- **Zero Control** → Agent 可能需要读/修改自身代码 → 语言认知复杂度必须低
- **Controllable Evolution** → 快速迭代探索正确抽象 → 编译/开发速度优先
- **Interface is Existence** → 并发调度是核心能力 → 并发模型必须自然
- **Transitional Structures** → 当前架构是脚手架 → 不需要终态级别的类型安全

### A3 Diagnose — 识别缺口

语言选择的判断标准不是"哪个语言最好"，而是"哪个语言在 Axis 的约束集下综合得分最高"。

结构性张力的判断标准不是"能否现在解决"，而是"是否封死了未来的解决路径"。

### A4 Realign — 映射到原则

- **阶段论**：不用终态理想批判当前阶段的妥协
- **主要矛盾优先**：当前主要矛盾是"快速探索正确抽象"，不是"编译期安全"或"极致性能"
- **对立统一**：迭代速度与类型安全是对立面，在当前阶段迭代速度是主要方面

### A5 Minimize — 最小单元

需要回答的最小问题：
1. Go 是否满足所有硬约束？→ 是
2. 是否有候选语言在满足硬约束的同时显著优于 Go？→ 否
3. 结构性张力是否由语言选择导致？→ 否（是设计哲学的内在矛盾）
4. 结构性张力是否封死了演化路径？→ 否（append-only log / gate / sandbox 保留了操作空间）

## Phase III: Sublation（扬弃）

### A7 Distill — 四栏复盘

| 保留 | 修正 | 剔除 | 沉淀 |
|------|------|------|------|
| Go 是当前阶段的最优选择 | "best"需限定为"阶段最优"而非"终态最优" | "Go 没有缺点"的暗示 | 语言选择的判据是约束集匹配度，不是语言本身的优劣 |
| 五个结构性张力是真实的 | 它们不是"缺陷"，是设计哲学的内在矛盾 | "需要现在解决"的紧迫感 | 这些张力属于更高阶段的矛盾，当前阶段的正确做法是不封死路径 |
| 实验价值独立于商业结果 | | | 判断力的积累是不可替代的收益 |
| 赌 AGI 演化速度是合理策略 | 需要显式承认这是一个赌注，不是确定性 | | 写入为显式的阶段论判断 |

### A8 Sublate — 写回规则

**沉淀为原则：**

1. **语言选择原则**：在约束集下做综合最优选择，不追求单一维度的极致。Axis 的约束集（单二进制 + 快速迭代 + 低认知复杂度 + 自然并发）指向 Go。

2. **结构性张力处理原则**：区分"当前阶段的缺陷"和"更高阶段的矛盾"。对后者，正确做法是保留操作空间（可观测性 + 显式 gate + 不可逆操作的审计），而非强行在当前阶段解决。

3. **演化赌注的显式化**：Axis 显式选择"赌 AI 能力的演化速度会解决当前无法优雅解决的问题"。这是一个阶段论判断：
   - 如果赌对了 → 结构性张力由更强的 Agent 从内部解决
   - 如果赌错了 → Axis 需要在"人类审批每次演化"的状态下长期存活
   - 对冲策略 → 保持系统可运行、有用户、有使用场景（活到那一天）

4. **不封死原则**：任何当前的架构决策都不应使未来的 Agent 无法重写该决策。具体表现：
   - Contract 可被扬弃（不是硬编码约束）
   - Event log 是 append-only（历史不可篡改但可被重新解释）
   - Sandbox 是隔离而非禁止（允许否定，但否定的结果需验证后才逃逸）

## 矛盾论总结

| 矛盾 | 主要方面（当前阶段） | 次要方面 | 处理方式 |
|------|---------------------|----------|----------|
| 迭代速度 vs 类型安全 | 迭代速度 | 类型安全 | 选 Go，接受运行时错误风险 |
| 人类审批 vs Agent 自治 | 人类审批 | Agent 自治 | 保留 gate，等待能力证明 |
| 当前可解 vs 未来可解 | 未来可解 | 当前可解 | 不封死路径，赌演化速度 |
| 商业价值 vs 实验价值 | 实验价值 | 商业价值 | 以判断力积累为首要收益 |
