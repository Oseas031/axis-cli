## Core Workflow: Spec-First Development

### The Philosophy

Axis is **spec-first**. Specs are implementation contracts, not decorative notes.

From `docs/architecture/spec-lifecycle-conventions.md`:

- Requirements define what must be true
- Design defines how Axis intends to satisfy requirements
- Tasks define execution order and status
- Code must not contradict active specs

### The Workflow

```text
1. Read workflow/entry.md (route your task)
2. Write requirements.md (what to build)
3. Write design.md (how to build it)
4. Write tasks.md (break into T1, T2, T3...)
5. Implement each T with tests
6. Run go test ./... after each T
7. Update docs/status/current-progress.md
8. Mark spec status: Draft → Planned → In Progress → Completed
```

### Why This Works for You

You don't need to know "how to code" every detail. You need to:

1. **Define clear requirements** - AI can implement from clear requirements
2. **Define clear boundaries** - AI respects semantic boundaries
3. **Define verification criteria** - AI can write tests to verify
4. **Review AI output** - Your design judgment catches issues

## Contract Design: Your Core Skill

### What is a Contract?

In Axis, a **Contract** is the **input/output shape** of an operation, not the execution logic.

From `docs/architecture/semantic-boundaries.md`:

- **AgentContract owns**: input/output shape and execution contract
- **AgentContract does NOT own**: scheduler policy, provider credentials, context retrieval

### Contract Structure

```go
type AgentContract struct {
    ContractID   string        // e.g., "self/analyze-change-request"
    InputSchema  *InputSchema  // Fields: Name, Type, Required, Description, Enum
    OutputSchema *OutputSchema // Fields: Name, Type, Required, Description, Enum
}
```

### Example: AnalyzeContract

```go
func AnalyzeContract() *types.AgentContract {
    return &types.AgentContract{
        ContractID: "self/analyze-change-request",
        InputSchema: &types.InputSchema{Fields: []types.FieldDef{
            {Name: "change_description", Type: types.FieldTypeString, Required: true},
            {Name: "target_files", Type: types.FieldTypeArray, Required: false},
            {Name: "motivation", Type: types.FieldTypeString, Required: false},
        }},
        OutputSchema: &types.OutputSchema{Fields: []types.FieldDef{
            {Name: "impact_scope", Type: types.FieldTypeArray, Required: true},
            {Name: "risk_level", Type: types.FieldTypeString, Required: true},
            {Name: "suggested_order", Type: types.FieldTypeArray, Required: false},
            {Name: "confidence", Type: types.FieldTypeFloat, Required: true},
        }},
    }
}
```

**Key observations**:
- Contract doesn't say "how to analyze" - only "what goes in, what comes out"
- Input fields have clear types and required flags
- Output fields are verifiable (risk_level as string, confidence as float)

### Contract Design Principles

#### 1. Boundary Clarity

Contract only defines shape, not implementation.

**Bad**:
```go
// This describes implementation, not contract
{Name: "analysis_method", Type: types.FieldTypeString, Required: true}
```

**Good**:
```go
// This describes input shape only
{Name: "change_description", Type: types.FieldTypeString, Required: true}
```

#### 2. Type Strictness

Use explicit types, avoid ambiguity.

**Bad**:
```go
{Name: "status", Type: types.FieldTypeString, Required: true} // What values?
```

**Good**:
```go
{Name: "is_acceptable", Type: types.FieldTypeBool, Required: true} // Clear
// OR
{Name: "risk_level", Type: types.FieldTypeString, Required: true, Enum: []string{"low", "medium", "high"}}
```

#### 3. Required Field Discipline

- **Input Required**: Conservative - only mark truly necessary fields
- **Output Required**: Strict - guarantee caller can depend on these fields

#### 4. Namespace Convention

ContractID uses namespaces to avoid conflicts:

```go
const (
    ContractIDAnalyze   = "self/analyze-change-request"  // Self-iteration
    ContractIDImplement = "self/implement-change"        // Self-iteration
    ContractIDValidate  = "self/run-validation"          // Self-iteration
)
```

For external contracts, use domain prefixes:
- `domain/feature-name`
- `memory/immunity-promote`
- `context/bundle-query`

#### 5. Minimalism

Only include necessary fields. Don't add "future-proof" fields.

**Bad**:
```go
InputSchema: &types.InputSchema{Fields: []types.FieldDef{
    {Name: "change_description", Type: types.FieldTypeString, Required: true},
    {Name: "target_files", Type: types.FieldTypeArray, Required: false},
    {Name: "motivation", Type: types.FieldTypeString, Required: false},
    {Name: "future_field_1", Type: types.FieldTypeString, Required: false}, // Why?
    {Name: "future_field_2", Type: types.FieldTypeString, Required: false}, // Why?
}},
```

**Good**:
```go
InputSchema: &types.InputSchema{Fields: []types.FieldDef{
    {Name: "change_description", Type: types.FieldTypeString, Required: true},
    {Name: "target_files", Type: types.FieldTypeArray, Required: false},
    {Name: "motivation", Type: types.FieldTypeString, Required: false},
}},
```

#### 6. Testability

Every contract should have schema validation tests.

```go
func TestAnalyzeContract_Schema(t *testing.T) {
    c := AnalyzeContract()
    if c.ContractID != ContractIDAnalyze {
        t.Errorf("expected contract ID %q, got %q", ContractIDAnalyze, c.ContractID)
    }
    if len(c.InputSchema.Fields) != 3 {
        t.Errorf("expected 3 input fields, got %d", len(c.InputSchema.Fields))
    }
}
```

#### 7. Evolution-Friendly

Contract changes follow evolution protocol:

- **Add field**: Backward compatible (same contract ID)
- **Change type**: Breaking change (new contract ID)
- **Remove field**: Breaking change (new contract ID)

### Contract Design Checklist

When designing a new contract:

- [ ] Does it define input/output shape only?
- [ ] Are types explicit (String, Int, Float, Bool, Array, Object)?
- [ ] Are required fields correctly marked?
- [ ] Does it use namespace in ContractID?
- [ ] Is it minimal (no unnecessary fields)?
- [ ] Does it have schema validation tests?
- [ ] Is it registered in AllContracts() and RegisterAll()?
- [ ] Is it documented in spec?

## AI Collaboration Best Practices

### Your Role vs AI Role

| Aspect | You | AI |
|--------|-----|-----|
| **Requirements** | Define what to build | Clarify ambiguities |
| **Design** | Define how (boundaries, contracts) | Fill in implementation details |
| **Implementation** | Review, approve | Write code, write tests |
| **Verification** | Define criteria | Run tests, report results |
| **Documentation** | Review, sync | Draft, update |

### Prompting Strategy

#### 1. Provide Context First

Don't say "write a function". Say:

```text
Context: Axis is an agent-native scheduling system.
Task: Implement immunity memory promotion.
Constraints: 
- Must follow semantic-boundaries.md
- Must use existing memory layer (no new store)
- Must be non-invasive (FR8 in requirements.md)
```

#### 2. Reference Existing Code

```text
Reference: internal/agent/contracts/analyze.go for contract structure
Reference: docs/architecture/semantic-boundaries.md for what memory layer must NOT do
```

#### 3. Ask for Verification

```text
After implementation, run:
- go test -race ./internal/memory/...
- staticcheck ./...
- gosec ./...
```

#### 4. Request Schema Tests

```text
Include schema validation tests like contracts_test.go
```

### Reviewing AI Output

#### What to Check

1. **Boundary violations** - Does code violate semantic-boundaries.md?
2. **Hidden dependencies** - Are external deps introduced unnecessarily?
3. **Test coverage** - Are there tests for error paths?
4. **Documentation sync** - Are docs updated?
5. **Spec compliance** - Does code match requirements.md?

#### What to Accept

AI-generated code is acceptable if:

- It follows CLAUDE.md constraints
- It passes go test ./...
- It respects semantic boundaries
- It has tests
- It's documented

You don't need to understand every line. You need to verify it follows the rules you designed.

## Decision Framework: When to Add Complexity

From `workflow/occams-razor-architecture-simplification.md`:

### Three Judgments

#### 1. Is It Required Now?

Implement only minimal capabilities for current milestone.

If idea belongs to future milestone:
- Write a spec first
- Do NOT mix into current implementation

#### 2. Is Existing Lightweight Solution Sufficient?

Before adding complexity, explain why existing solution is insufficient:

- **UI**: Why is CLI/shell not enough?
- **Model Provider**: Why is MockProvider not enough?
- **Workflow**: Why is existing routing in entry.md not enough?
- **Automation**: Why are non-blocking reminders not enough?
- **Dependencies**: Why is stdlib not enough?
- **Persistence**: Why is file-backed state not enough?

**If you cannot explain, do NOT add by default.**

#### 3. Does It Break Scaffold-to-Self?

Remember: workflow, contract, permission rule, spec are **transitional structures**.

New rules must not disguise temporary scaffolding as permanent control.

## Common Pitfalls and How to Avoid

### Pitfall 1: Skipping Spec

**Symptom**: Directly coding without requirements.md/design.md/tasks.md

**Fix**: Always follow spec-first protocol. Read workflow/entry.md first.

### Pitfall 2: Vague Requirements

**Symptom**: requirements.md says "implement memory system" without FR1, FR2, FR3...

**Fix**: Use Functional Requirements (FR1, FR2, FR3...) with clear acceptance criteria.

Example from immunity-memory requirements.md:

```markdown
### FR1: Immunity record schema
An Immunity record MUST contain:
- `immunity_id` — stable identifier
- `source_task_id` — the failed task this was promoted from
- `signature` — a deterministic hash of {...}
...
```

### Pitfall 3: Contract Contains Implementation

**Symptom**: Contract has fields like "implementation_method" or "algorithm"

**Fix**: Contract only defines input/output shape. Implementation is separate.

### Pitfall 4: No Tests

**Symptom**: Code works but no tests

**Fix**: Every implementation must have tests. Ask AI to write tests first.

### Pitfall 5: Boundary Violation

**Symptom**: Memory layer calls scheduler directly

**Fix**: Check semantic-boundaries.md. Each module has "Must NOT do" list.

### Pitfall 6: Missing Documentation Sync

**Symptom**: Code done, docs not updated

**Fix**: Always sync 4 docs after work (see Daily Development Workflow).

## Verification Standards

### Machine-Checkable Criteria

From spec-lifecycle-conventions.md, good verification criteria are:

| Good (machine-checkable) | Bad (subjective) |
|---|---|
| `go test -race ./internal/memory/... passes` | "Code looks clean" |
| `go list -deps ./internal/kernel/... excludes internal/memory/ledger` | "Doesn't seem to leak abstraction" |
| `axis context preview --no-flag output byte-identical to golden file` | "Backward compatible" |
| Event log contains exactly one `memory.immunity.promoted` per CLI invocation | "Works as expected" |
| `gosec ./... reports zero issues` | "Probably secure enough" |

**Your spec acceptance criteria must use "Good" column language.**

### Build and Test Commands

From CLAUDE.md:

```bash
go build -o axis-dev.exe ./cmd/axis   # Windows dev binary
go test -race ./...                    # must pass before any commit
gofmt -w . && go vet ./...            # formatting + vet
staticcheck ./... && gosec ./...       # static analysis + security
```
